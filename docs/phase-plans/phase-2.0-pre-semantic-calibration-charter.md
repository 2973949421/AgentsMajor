# Phase 2.0-pre 赛事语义校准临时高优先级纲领

## 1. 文档定位

这份文档是 `Phase 2.0-pre` 的临时高优先级纲领。

它的作用不是替代长期母文稿，而是在当前阶段先统一一个最高优先级口径，避免“规则层、materials 层、runtime 层、前端层各说各话”。

在 `Phase 2.0-pre` 完成前，本文件高于分散在旧文档中的“直接进入 16 队 bracket / 赛事调度”的旧阶段判断。等单图和定制 BO3 校准完成后，再把稳定内容回写进长期母文稿。

## 2. 阶段定位

`Phase 2.0-pre` 不是完整 16 队 bracket 阶段。

它是完整赛事前的语义校准阶段，目标是先把比赛真正要比的东西跑顺：

```text
地图命题
队伍初始方案
coach / player / judge 分工
方案攻防协议
赛后方案修正机制
前端对方案攻防的表达
```

当前阶段如果直接扩到 16 队，只会把尚未校准的语义噪声放大到整届赛事里。

## 3. 当前唯一主线目标

当前主线按固定顺序推进：

```text
1. 先真正跑通一张精心设计的地图。
2. 再基于同一套规则跑通一场精心设计的 BO3。
3. 单图和定制 BO3 都通过后，再进入完整 16 队正式扩张。
```

这里的“跑通”不是命令执行完毕，而是赛事语义真正成立。

## 4. 比赛本质定义

Agent Major 当前阶段不是单纯的 AI 打 CS，也不是单纯的文本辩论赛。

它的核心定义是：

```text
两套完整项目方案
在 CS 赛制和攻防语言下
进行逐回合对抗、逐图校验、逐场修正
最终沉淀出更强方案的赛事系统
```

当前地图和回合不是只在比“谁更会说”，而是在比：

```text
机会判断
MVP 设计
工程交付
商业闭环
增长运营
规模化治理
长期壁垒
```

## 5. 地图命题要求

单图不是抽象地图，而是明确项目命题图。

每张进入 `Phase 2.0-pre` 的地图都必须先定义清楚：

```text
这张图在比什么。
攻方主要在攻击对手方案的哪类部分。
守方主要在保卫己方方案的哪类部分。
什么样的进攻算有效。
什么样的防守算有效。
什么样的回合胜势会累计成整图胜势。
```

地图命题必须能被：

```text
队伍作为入场初始方案理解和携带。
coach 在允许窗口内读取并修正。
player 按角色分工执行。
judge 用来裁定回合和整图胜负。
frontend 用来展示当前攻防焦点。
```

## 6. 队伍参赛资产要求

进入 `Phase 2.0-pre` 的两支队伍，不是只带 5 名选手和 1 名教练入场。

它们必须各自带着一份 `初始方案包` 入场。

初始方案包至少包括：

```text
机会判断
MVP 主张
工程交付思路
商业闭环思路
增长运营思路
规模化治理思路
长期壁垒思路
```

当前阶段不要求 7 个部分平均展开，但必须明确：

```text
当前地图重点调用哪一部分。
哪些部分是本图核心胜负点。
哪些部分是本图可牺牲项。
哪些部分被打穿后会直接导致输图。
```

## 7. 角色职责要求

### coach

coach 负责：

```text
不替队伍在局前发明总纲。
只在暂停、半场换边或图后做方案修正。
根据上一阶段暴露出的漏洞重排后续攻防重点。
赛后形成方案修正意见。
```

### players

player 负责：

```text
按角色切分方案执行任务。
不是人人重复整套方案。
而是各自承担不同攻防职责。
```

当前角色原则：

```text
IGL：组织结构、节奏、资源分配。
Entry：先手切口、打开局面。
AWPer：高精度关键点压制。
Star Rifler：核心输出、关键回合 carry。
Lurker：抓漏洞、侧翼突破、反制。
Support / Anchor：补全、修复、稳住关键节点。
```

### judge

judge 负责：

```text
判断当前攻防是否有效。
判断谁更成功打击了对手方案要害。
判断谁更成功守住了己方方案核心。
判断胜负依据是否贴合当前地图命题。
```

judge 不是文风评审，也不是单纯比分播报器。

## 8. 攻防协议

当前攻防定义固定为：

```text
进攻方：持续寻找并攻击对手方案中的漏洞、脆弱点和不成立处。
防守方：根据战术和经济布置，优先保卫己方方案中最关键、最不可失守的部分。
```

因此每个回合都必须能回答：

```text
谁在攻。
谁在守。
攻方在攻什么。
守方在守什么。
为什么要这样分配资源。
```

战术布置和经济布置必须服务于方案攻防，不允许脱离方案语义单独存在。

## 9. 赛后方案修正机制

队伍不是静态方案，而是带版本号持续进化的参赛体。

每张图结束后，两支队伍都必须形成一份 `图后方案修正结果`，至少包括：

```text
本图被打穿的地方。
本图最有效的进攻路径。
本图最有效的防守方式。
哪些观点需要保留。
哪些观点需要修正。
下一图 / 下一场准备如何升级方案。
```

完整赛事的冠军方案不是初始稿，而是多场对抗后的 surviving version。

## 10. 前端展示要求

当前前端不是只播比分，而是要让观众看懂方案攻防。

`Phase 2.0-pre` 前端至少要逐步能表达：

```text
当前地图命题。
当前回合攻守关系。
当前主攻点。
当前主守点。
当前比分和运行状态。
当前回合裁判结论。
图后方案修正结果。
```

如果前端只能看出“谁赢了”，却看不出“为什么在方案层赢了”，则当前阶段不算通过。

## 11. 当前阶段交付物

`Phase 2.0-pre` 第一批必须产出以下文稿和维护资产：

```text
临时高优先级纲领。
单图命题稿。
双队初始方案稿。
coach / player / judge 职责协议。
裁判评分规程。
前端展示字段清单。
图后方案修正模板。
定制 BO3 验收标准。
```

这些资产优先于继续扩张队伍数量、继续泛化 prompt 或继续做外围内容。

## 12. 当前阶段非目标

`Phase 2.0-pre` 明确不做：

```text
不直接展开完整 16 队正式赛事。
不优先做完整 bracket 系统。
不优先扩更多队伍资产。
不优先做完整新闻站、奖项站和生态站。
不优先继续扩真实 LLM 范围。
不把模型调用成功误判为赛事语义成功。
```

## 13. 当前阶段通过标准

`Phase 2.0-pre` 的通过标准不是工程命令跑完，而是以下条件成立：

```text
单图命题被真实送入比赛链路。
双方初始方案被真实送入比赛链路。
攻防围绕方案漏洞与方案核心展开。
裁判按明确规程给出可解释裁决。
前端能看懂当前攻守与方案焦点。
图后能沉淀方案修正结果。
在此基础上，定制 BO3 也能稳定复现同样的赛事语义。
```

如果这些条件没过，就不进入完整 16 队 bracket 扩张。

## 14. 与长期母文稿的关系

当前阶段的维护顺序固定为：

```text
先维护本临时高优先级纲领。
再维护与当前校准直接相关的单图 / 双队 / 裁判 / 前端文稿。
等单图与定制 BO3 跑顺后，再把稳定内容回写进长期母文稿。
```

因此，当前项目主线应表述为：

```text
先执行 Phase 2.0-pre：
单图赛事语义校准 -> 定制 BO3 校准 -> 再进入 16 队正式扩张
```
