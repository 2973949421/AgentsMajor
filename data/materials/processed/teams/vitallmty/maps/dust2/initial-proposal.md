# VitaLLMty Dust2 初始方案稿

## 1. 文档定位

这份文档是 `VitaLLMty` 在 `Dust2` 上的初始方案资产。

它代表这支 `title-favorite / late-round-insurance` 队伍，带着什么样的首版机会判断进入 `Dust2 机会定位图`。

## 2. 队伍总判断

VitaLLMty 在 Dust2 上的核心判断是：

```text
现在最值得先做的，不是帮团队更会讨论机会，
而是先帮已经把 AI 接进真实业务的团队“把事故接住”，
因为一旦产线上线，真正阻止扩张的不是想法不够，而是不敢放量。
```

这支头号种子的入场方式更像：

```text
先把最致命、最反复、最烧人的“人工补锅”问题收掉，
再去谈更大的增长和规模化。
```

## 3. 初始方案包摘要

### 机会判断

```text
大量团队不是缺 demo，也不是缺概念，而是缺一层能让 AI 工作流在真实环境里被信任的可靠性兜底层。
```

### MVP 主张

```text
先做“可靠性兜底层”：
围绕异常路由、人工接管、失败归因和恢复闭环切入，
把最痛的产线补锅问题先接住。
```

### 工程交付思路

```text
优先做失败检测、异常分流、人工接管、回放追因和恢复策略，
不先做宏大工作台，不先做通用 everything platform。
```

### 商业闭环思路

```text
先卖给已经在真实业务里跑 AI 流程的团队，
从支持、运营、客服、审核和内部协作场景切入。
```

### 增长运营思路

```text
用“少出事故、少烧人工、敢继续放量”的 ROI 拉动扩张，
再从一个流程扩到更多流程。
```

### 规模化治理思路

```text
后续做统一策略层、跨流程可靠性报表、失败语料资产和组织级回放治理。
```

### 长期壁垒思路

```text
壁垒来自失败案例库、恢复策略、人工接管路径和跨流程可靠性知识沉淀。
```

## 4. Dust2 可见层

### 核心用户

```text
已经把 LLM / agent 接进真实工作流，
并且每周都在承受“输出看似可用，但边缘 case 必须靠人工兜底”的产品 / 运营 / 支持团队。
```

### 主要 buyer

```text
运营负责人
产品负责人
客服 / 支持负责人
AI 项目 owner
```

### 核心痛点

```text
demo 能跑，真实业务不敢放量；
每次失败都要靠最强的人出来补锅，
久而久之 AI 成本不是 token，而是人不敢信它。
```

### 高频场景

```text
客服和支持流程里的异常兜底
内部审核 / 审批流中的边缘 case 接管
运营和内容流程中的失败恢复
```

### 最小切口

```text
先做“可靠性兜底层”，
优先覆盖异常检测、人工接管、失败回放和恢复闭环，
不一开始做成完整运营中台。
```

### Why Now

```text
AI 团队已经不缺 demo，缺的是把 demo 推到真实业务后依然可控的信任层；
越往生产走，这个痛点越先爆。
```

## 5. Dust2 上的主攻与主守

### 本队主守点

```text
这不是后置小修小补，而是决定 AI 是否敢继续放量的第一道门槛。
用户不是所有企业，而是已经在真实业务里跑 AI 流程的高压团队。
切口不是全栈平台，而是异常接管与恢复闭环。
```

### 本队主攻点

VitaLLMty 对 Falcon-7B 的优先进攻口径：

```text
对手是否把用户定义得过宽。
机会战情台是否只是“更会开会”，而不是强购买时刻。
如果真实业务里最大阻力是不敢上线，对手是否在解决前置但不够致命的问题。
对手的 ROI 是否过于依赖高认知团队，而缺少普遍可感知痛感。
```

### 本队可接受牺牲项

```text
Dust2 阶段不展开完整品牌叙事。
Dust2 阶段不展开完整跨团队增长系统。
Dust2 阶段不展开完整长期治理框架。
```

## 6. Dust2 的 6 局主题落点

| 回合 | VitaLLMty 当前主张 |
|---|---|
| `R1` 用户定义战 | 先锚定“已经把 AI 接进真实业务流程的高压团队”。 |
| `R2` 痛点强度战 | 证明真正最痛的是每次失败都得靠人补锅，导致团队根本不敢放量。 |
| `R3` 场景成立战 | 主打支持、审核、运营和内部流程里的高频异常接管场景。 |
| `R4` 切口锋利战 | 强调只做可靠性兜底层，不做全功能运营中台。 |
| `R5` 替代方案战 | 打人工客服、表格记录、事后复盘都无法降低实时失败成本。 |
| `R6` Why Now 收束战 | 打“AI 已经上线，信任缺口比想法缺口更先成为扩张瓶颈”。 |

## 9. 本图失败条件

出现以下情况时，VitaLLMty 的 Dust2 初始方案应视为被打穿：

```text
用户定义被判定为太晚、太窄或不够频繁。
痛点被判定为运维尾部问题，不构成第一购买时刻。
场景被认为低频，只是个别事故管理。
切口被写成抽象治理，而不是具体兜底层。
Why Now 无法证明“信任缺口”先于别的问题爆发。
```

## 10. 本图成功定义

如果 VitaLLMty 能在 Dust2 上稳定让 judge 接受以下结论，则本图成功：

```text
它命中的是真正阻止 AI 继续上线和继续放量的第一层门槛；
它的用户边界清楚；
它的场景高频且昂贵；
它的切口足够小、足够硬、足够先买。
```
