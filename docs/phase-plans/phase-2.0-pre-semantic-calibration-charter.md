# Phase 2.0-pre 语义校准纲领

## 1. 文档定位

这份文档是 `Phase 2.0-pre` 的当前最高口径，用来统一：

- 规则层
- materials 层
- runtime 层
- 前端表达层

在 `Phase 2.0-pre` 收口前，本文件高于旧阶段里分散的局部判断。

## 2. 当前阶段目标

`Phase 2.0-pre` 不追求直接扩到 16 队完整赛事，而是先把赛事语义内核校准好。

当前主线顺序固定为：

1. 跑通单图语义校准
2. 跑通定制 BO3
3. 再进入 16 队正式扩展

## 3. 比赛本质

Agent Major 当前不是普通 FPS 模拟，也不是单纯文本辩论。

它的本质是：

- 两套完整项目方案
- 在 CS 攻防语言与赛制中逐回合对抗
- 由裁判按地图命题与判规裁决
- 最终沉淀出 surviving version

## 4. 地图角色

地图不是给队伍再写一份新方案。

地图只负责：

- 提出命题
- 规定攻守焦点
- 规定裁判轴
- 提供区域语义

因此地图的职责是挑战方案，不是创造方案。

## 5. 队伍唯一方案

从 `6.1` 开始，队伍方案口径固定如下：

- 每支队伍只有一份唯一方案
- 唯一方案文件固定为：
  - `processed/teams/<team-slug>/initial-proposal.md`
  - `processed/teams/<team-slug>/initial-proposal.json`
- `initial-proposal.json` 是唯一 runtime 方案资产
- `initial-proposal.md` 是人工阅读与维护稿

本阶段不再保留并列的 `strategy` 方案层，也不再保留地图级队伍方案真相。

## 6. 角色职责

### coach

coach 不在局前发明新总纲，只负责：

- 暂停修正
- 整场 BO3 赛后复盘

### players

players 按角色职责执行，不重复整队总纲。每名选手只承担自己在方案中的那部分攻防职责。

### judge

judge 负责判断：

- 谁打中了对手方案缺口
- 谁守住了己方核心成立点
- 这个结论是否贴合当前地图命题与裁判轴

judge 不是文风评审，也不是比分播报器。

裁判可审计契约以 `docs/phase-plans/phase-2.0-pre-judge-audit-contract.md` 为准。真实 LLM 裁判回合必须产出完整 `judgeDiagnostic`，并把它作为前端展示、coach 暂停修正和 BO3 赛后复盘的主证据来源。

## 7. 攻防协议

当前攻防定义固定为：

- 进攻方：持续攻击对手方案中的漏洞、脆弱点与不成立处
- 防守方：优先保卫己方方案中最不可失守的核心部分

每个回合都必须能回答：

- 谁在攻
- 谁在守
- 攻方在攻什么
- 守方在守什么
- 为什么这样分配资源

## 8. BO3 赛后复盘

队伍不是静态方案，而是带版本号持续演化的参赛体。

每场定制 BO3 结束后，队伍都应该产出赛后复盘结果，至少回答：

- 本场被打穿的部分
- 本场最有效的进攻路径
- 本场最有效的防守方式
- 哪些观点要保留
- 哪些观点要修正

## 9. 前端表达

前端不能只播比分，必须能让观众看懂方案攻防。

当前最少要逐步稳定表达：

- 当前地图命题
- 当前回合攻守关系
- 当前主攻点
- 当前主守点
- 当前裁判结论
- 当前队伍方案摘要

## 10. 当前收口标准

`Phase 2.0-pre` 是否通过，不看命令是否跑完，而看这些条件是否成立：

1. 地图命题真正进入比赛链路
2. 队伍唯一方案真正进入比赛链路
3. 攻防围绕方案漏洞与核心展开
4. 裁判能给出可解释、可复盘、可被系统审计的结构化裁决
5. 前端能看懂当前方案攻防焦点
6. 单图与定制 BO3 都能复现同样的赛事语义

未满足这些条件前，不进入完整 16 队 bracket 扩展。
