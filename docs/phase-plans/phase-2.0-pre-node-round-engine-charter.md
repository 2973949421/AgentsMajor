# Phase 2.0-pre 节点化回合引擎纲领

## 1. 文档定位

这份文档用于冻结当前关于“节点化回合引擎”的共识，避免后续实现、调试或新对话中发生记忆漂移。

它不是立即执行计划，也不是完整技术设计。它是后续计划、schema、materials、runtime 和前端表达的纲领依据。

本纲领建立在现有 `Phase 2.0-pre` 主线之上：

- `engine.ts` 已经从巨型业务文件重构为回合编排层。
- `JudgePipeline`、`Economy/Output`、`CoachService` 已经拆出第一版结构。
- 当前下一步不应继续把裁判和经济逻辑塞回 `engine.ts`。
- 后续要从“整回合裁判制”转向“节点状态推进制”。

## 2. 核心判断

Agent Major 的本质不是纯 CS 模拟器，也不是普通商业文本辩论。

它的本质是：

```text
以 CS 地图、时间、交火、经济和胜负规则作为对抗骨架；
以商业计划、产品分工、运营分工、技术分工和组织协作为语义内核；
让 agent 在节点化战术空间里推进、碰撞、验证和暴露方案。
```

因此，后续设计必须同时保留两条线：

- CS 线：地图空间、时间阶段、控图、交火、进点、下包、守包、拆包、全歼、经济继承。
- 商业线：市场切入、渠道控制、产品验证、运营转化、技术支撑、组织协同、资源调度、方案漏洞。

任何只保留 CS 动作、丢掉商业底色的实现都是偏离。

任何只保留商业辩论、丢掉 CS 位置和胜负条件的实现也是偏离。

## 3. 当前要解决的结构问题

现有回合链路更接近：

```text
team_plan
-> agent_action
-> judge 判断整回合胜负
-> combat / broadcast 根据胜负补回放
```

这个结构已经能稳定生成，但存在根本限制：

- winner 容易先被裁判锁定，后续事实变成解释 winner。
- combat 更像胜负后的回放合成，不是独立因果链。
- 经济、地图、道具、人数、信息差很难自然积累成胜负。
- proof score、scorecard、judge reason 容易后验合理化。
- 攻守偏置和模板比分只能诊断，难以从结构上消化。

节点化回合引擎要把链路改为：

```text
地图节点状态
-> 时间点行动
-> 局部裁判
-> 状态更新
-> 胜负条件检查
-> 下一时间点
-> 最终回合事实
```

裁判不再是整回合赢家制造者，而是局部事实裁定者和语义解释者。

## 4. 地图节点原则

地图节点不能再只是字符串或 enum。

地图节点必须成为状态容器。

每个节点至少应能记录：

- 节点 id 与显示名称。
- 所属战区。
- 可连接节点。
- 攻守默认优势。
- 常见交火方向。
- 当前攻方人数。
- 当前守方人数。
- 当前控制权。
- 当前可见信息。
- 当前道具影响。
- 当前商业计划意图。
- 当前风险。
- 交火前状态。
- 交火后状态。
- 信息是否传递给队友。
- 本节点对下一时间点的影响。

节点不是为了限制 agent 死走固定路线，而是为了让输出有空间、有位置、有状态。

## 5. Dust2 第一版节点库

Dust2 第一版应尽量按真实地图结构建立节点库。第一版不要求每回合激活所有节点，但节点库要先设计完整，避免后续补节点时反复改 schema 和 prompt。

节点库不能只列点名，还必须固定连接关系。Dust2 节点 graph 的正式资产为：

```text
data/materials/processed/maps/dust2/node-graph.json
data/materials/processed/maps/dust2/node-graph.md
```

后续 LLM prompt 和 runtime 不能让模型自行猜测从一个地点到另一个地点的路线。所有移动、转点、回防、夹击都应基于 node graph 的节点和边。

node graph 还必须记录阶段可达性。仅知道“两个节点相连”不够，系统还要知道：

- 当前时间点最远能到哪里。
- T 方和 CT 方到同一节点的先后差异。
- CT 方开局天然更接近 A 点、B 点、中门、A 小、B 门、B 窗等防守位置。
- T 方默认展开阶段不能直接到达包点或下包。
- 下包、拆包、守包、包炸只能发生在合理的后续时间点。

因此，后续 prompt、normalizer 或 validator 应能拒绝或修正“默认阶段 T 已经站进包点”这类不合理事实。

### 5.5 第一版使用方式

第一版每回合不需要把所有节点都送给 LLM。

推荐策略：

- 全量节点库由代码和 materials 保存。
- 每回合根据 team plan、spawn、买型、上一回合历史选出 3-5 个活跃节点。
- 时间点推进时只更新被激活节点和受影响连接节点。
- 未激活节点保持默认状态或轻量状态。

## 6. 回合时间点

一回合建议拆成 3-5 个时间点。时间点不是固定必须全部走完，而是根据胜负条件提前结束。

推荐阶段：

1. 默认展开
   - 开局站位、默认控图、经济买型、商业计划开局部署。

2. 第一接触
   - 中路对枪、A 大争夺、B 洞压制、信息交换。

3. 中盘决策
   - 转点、夹击、假打、重新集合、保留道具、重新分配资源。

4. 进点 / 回防
   - 爆 A、爆 B、CT 回防、T 守点前部署。

5. 下包后 / 残局
   - 守包、拆包、残局、保枪、全歼。

胜负发生时间不固定：

- 最早可以在第一接触阶段因一方全歼结束。
- 中间可以在进点阶段因包点清空或回防失败进入结算。
- 最晚可以在包爆、拆包、守包全歼或时间耗尽时结束。

## 7. 胜负条件

节点化并不改变 CS 胜负条件。

代码应保留硬胜负条件：

- T 方全歼 CT 方，且回合状态允许判定为进攻胜。
- CT 方全歼 T 方，且炸弹未进入不可逆爆炸结算。
- T 方成功下包并守到爆炸。
- CT 方成功拆包。
- 时间耗尽且 T 方未下包。

局部裁判不能随意改写这些硬条件。

局部裁判只能裁定：

- 谁拿到信息。
- 谁被压退。
- 谁控制节点。
- 是否发生交火。
- 谁在局部交换中占优。
- 哪个商业计划被验证。
- 哪个商业计划被击穿。
- 哪个队伍获得下一时间点主动权。

最终回合胜负必须由代码根据状态机和硬条件落地。

## 8. Agent 行动模板

Agent 输出必须继续有商业底色，但要用模板约束稳定性。

每个 agent 在每个时间点不一定都要产生新动作。“按兵不动”是合法行动。

### 8.1 必填字段

- 当前所在节点。
- 目标节点。
- 行动类型。
- 商业意图。
- 配合对象。
- 风险判断。
- 预期结果。

### 8.2 可选字段

- 道具使用。
- 对枪目标。
- 转点条件。
- 撤退条件。
- 信息传递。
- 备用方案。
- 是否保持位置。

### 8.3 合法行动类型

- 保持位置。
- 架枪。
- 控图。
- 抢信息。
- 丢道具。
- 寻找对枪机会。
- 补枪。
- 假打。
- 转点。
- 进点。
- 回防。
- 守包。
- 拆包。
- 保枪。

### 8.4 按兵不动的合法表达

CS 里某个小区防守者可能在 `1:50-1:05` 之间都不换位置，也不发生新动作。

这不应被视为“没输出”。

它应被表达为：

- 保持位置。
- 持续架枪。
- 等待队友信息。
- 维持交叉火力。
- 保留道具。
- 观察对方是否转点。

## 9. 商业底色映射

每个 CS 行动都应允许映射到商业计划语义，但不能把商业语义变成空洞贴皮。

示例映射：

- 去中路抢信息：验证市场窗口、争夺渠道入口、测试对方响应速度。
- B 洞集合慢打：集中资源打穿对方薄弱业务线。
- A 大假打转 B：用产品声量吸引对手防守，再切入真实转化路径。
- CT 回防失败：组织响应慢、跨部门协作断层、资源调度不足。
- 保枪：承认当前窗口不值得继续投入，保留资源进入下一轮。
- 守包：把已取得的市场入口转化为可持续阵地。
- 拆包：在对方完成阶段性落地后，通过执行力和组织协同逆转。

后续 prompt 和 schema 应要求 agent 输出同时回答：

```text
我在 CS 空间里做什么？
我在商业计划对抗里验证什么？
我和队友的产品/运营/技术分工如何协作？
如果失败，暴露的是哪类方案问题？
```

## 10. 经济信息原则

经济信息应按以下口径固定：

```text
队伍共享经济信息；
个人持有具体经济资产；
队伍可以讨论本回合买型和发枪策略；
每名 agent 只能花自己的钱或接收队友发枪；
经济数据以 round 为单位继承；
经济状态持续影响开局购买、节点选择、道具可用性、交火风险和整体战术。
```

这意味着经济不是一个只给 judge 加权的标签。

经济应影响：

- 谁能买长枪。
- 谁能起 AWP。
- 谁能带关键道具。
- 谁适合承担首接触风险。
- 谁应该保枪。
- 队伍是否适合默认控图、快速进点、慢打、强起赌点或半起偷经济。

但经济不能直接硬控 winner。

## 11. 行动点数原则

节点化回合引擎需要一个轻量运行约束，连接地图时空、agent 小行动、经济状态和商业计划执行成本。

本阶段采用类似战棋或回合制战术游戏的行动点数模型，但只用于约束行动合法性和执行复杂度，不用于直接决定对枪胜负。

行动点数的职责：

- 限制每个时间点能走多远。
- 限制同一时间点能否同时移动、丢道具、抢信息、架枪或转点。
- 表达复杂商业计划的执行成本。
- 让低经济、低道具或组织受压的队伍必须取舍。
- 为地图时空关系提供可运行基础。

行动点数不负责：

- 不直接判定谁打赢枪。
- 不直接决定 winner。
- 不替代经济系统。
- 不模拟精确秒数、弹道、HP 或护甲。

第一版建议：

```text
每名 agent 每个时间点有基础 3 AP。
保持位置 / 等待信息：0-1 AP。
移动 1 条边：1 AP。
架枪 / 守角：1 AP。
抢信息：1 AP。
丢道具：1 AP。
补枪准备：1 AP。
进点：2 AP。
下包 / 拆包：2-3 AP。
保枪撤退：1-2 AP。
```

这些数值是第一版工程约束，不是最终平衡参数。

经济对行动点数的影响不应是简单“有钱多 AP，没钱少 AP”。更合理的方向是：

- 基础 AP 大体稳定。
- 经济影响动作质量、动作种类、道具可用性和风险。
- ECO 仍可抱团赌点、抢信息、偷 timing，但不能写成完整烟闪火爆弹。
- 全起可以执行更完整的多线默认和进点组合，但仍受 AP 限制。

商业计划也应消耗行动能力。一个方案如果同时要求产品、运营、技术、商业化多线推进，就必须在节点行动中体现资源压力和取舍。

## 12. 经济系统在节点化引擎中的位置

当前经济系统不推倒，也不原样作为裁判 proof 的隐性放大器。

经济系统在节点化引擎中的新位置是：

```text
经济状态
-> 购买 / 道具 / 武器 / 发枪 / 保枪能力
-> agent 可用动作和行动质量
-> 节点推进和局部事实
-> 胜负条件结算
```

旧路径应逐步弱化：

```text
经济差
-> 输出裁剪
-> judge 看到证据更少
-> judge 更容易判输
```

后续实现应保留：

- 队伍共享经济信息。
- agent 个人持有经济资产。
- 每回合经济继承。
- 队内可讨论买型和发枪。
- 经济影响购买、道具、风险、保枪和战术复杂度。

后续实现应避免：

- 经济直接决定商业计划是否成立。
- 经济直接让所有 scorecard 维度天然低分。
- 经济通过裁剪文本制造不可审计雪球。
- 经济成为隐藏控分机制。

## 13. 代码与 LLM 分工

代码负责：

- 地图节点状态。
- 人数与生死状态。
- 经济继承。
- 炸弹状态。
- 胜负条件检查。
- 时间点推进。
- 状态合法性校验。
- 不可能状态拦截。
- 最终事实落库。

LLM 负责：

- 选择合理行动。
- 解释局部战术意图。
- 判断商业计划碰撞。
- 给出局部交火理由。
- 解释为什么某个节点状态发生变化。
- 生成观赛叙事。

LLM 不能直接绕过代码写最终事实。

## 14. 局部裁判原则

每个时间点可以有一次局部裁判，但局部裁判应被约束为“节点事实裁定”，不是“整局赢家裁定”。

局部裁判输入：

- 当前时间点。
- 当前激活节点状态。
- 上一时间点状态变化。
- 双方行动模板输出。
- 经济与道具约束。
- 商业计划目标。
- 已知信息边界。

局部裁判输出：

- 节点控制权变化。
- 局部信息优势变化。
- 交火是否发生。
- 可能击杀或换人结果。
- 道具与资源消耗。
- 商业计划验证 / 破坏说明。
- 下一时间点主动权。
- 是否触发硬胜负条件检查。

局部裁判不得：

- 忽略代码状态强行判 winner。
- 把未发生的节点事实写成已发生。
- 把商业判词当作击杀事实。
- 为了剧情精彩制造不合法状态。

## 15. 成本与调用策略

根据当前 OpenCode Go 使用截图，`deepseek-v4-flash` 单次调用大致处于：

- 输入约 3000-8000 tokens。
- 输出约 500-3600 tokens。
- 单次成本约 0.0005-0.0015 美元。

当前月额度约 60 美元，足够支持节点化实验。

但节点化不能无脑增加 request 数。

推荐第一版调用策略：

- 每个时间点：一次双队行动生成，或双方各一次行动生成。
- 每个时间点：一次局部裁判。
- 每回合：3-5 个时间点。
- 每回合：一次最终摘要。

第一版目标调用量：

```text
每回合约 8-16 次 LLM 调用。
```

优化原则：

- 不让每个 agent 每个时间点都单独调用一次。
- 按兵不动 agent 由模板或代码轻量继承。
- 只让关键 agent 在关键节点展开细节。
- 保留 checkpoint 和断点续跑。
- 保留 reasoning，但最终 JSON 输出必须短、稳、可校验。

## 17. 第一版落地边界

第一版不追求完整真实 CS 物理模拟。

第一版应该做：

- 建立 Dust2 节点库。
- 建立节点状态容器。
- 建立 3-5 时间点推进结构。
- 建立 agent 行动模板。
- 建立局部裁判输出契约。
- 建立代码胜负条件检查。
- 保留商业底色。
- 保留经济继承。
- 保留断点续跑和 artifacts。

第一版不做：

- 不做完整枪械弹道模拟。
- 不做 HP / 护甲 / 每颗道具的物理效果。
- 不做每秒级移动模拟。
- 不做所有 agent 每节点独立 LLM 调用。
- 不硬控比分。
- 不写队伍专属补偿。
- 不让导演组直接改 winner。

## 18. 后续实现建议顺序

建议后续按以下顺序推进：

1. 文档和 materials 层
   - 固化 Dust2 节点库。
   - 给每个节点补背景、连接、攻守优势和商业语义。

2. Schema 层
   - 定义 `MapNodeState`。
   - 定义 `RoundPhaseState`。
   - 定义 `AgentPhaseAction`。
   - 定义 `LocalNodeVerdict`。

3. Runtime 层
   - 新增节点状态初始化。
   - 新增时间点推进器。
   - 新增局部裁判 pipeline。
   - 新增硬胜负条件检查。

4. 兼容层
